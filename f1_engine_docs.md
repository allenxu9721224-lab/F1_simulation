# F1 模拟引擎 (f1_engine.py) 核心逻辑文档

## 1. 架构概述 (Architecture Overview)

目前 `f1_engine.py` 已重构为一个**事件驱动的状态机 (Event-Driven State Machine)**。它打破了传统脚本“一次性跑完所有圈数”的模式，采用“静默模拟，按需暂停”的机制。
引擎会在后台逐圈计算当前赛道状况和所有车辆表现，仅在**玩家车手**遇到关键决策时刻（进站、安全车出动、天气突变）时打断循环并暂停，将控制权和提示信息（Event Prompt）交还给外部 API 等待玩家输入。

---

## 2. 核心类与方法 (Core Engine Classes)

### A. `Driver` (车手类)
负责管理单一车手的状态、圈速计算与进站行为。玩家与 AI 车手均基于此基类。

- **核心属性**：`base_alt` (基础圈速)、`pit_skill` (车队进站效率)、`rain_ability` (雨战能力评分)、`is_player` (是否为玩家控制)。
- **`calculate_lap_time` (单圈模拟核心)**：计算车手每一圈的确切耗时。受以下四大动态因素影响：
  1. **燃油消耗增益**：赛车随圈数增加而变轻，圈速会稳步下降（每圈提速约 0.06 秒）。
  2. **轮胎磨损衰减 (Tire Degradation & Cliff)**：引入了真实世界的 "Tire Cliff"（断崖式衰减）机制。一旦磨损超过对应的使用限界（如软胎 15 圈、硬胎 40 圈），圈速将受到极其严厉的指数级时间惩罚。
  3. **天气/轮胎惩罚 (Mismatch Penalty)**：如果在干地强行使用雨胎，或在湿地坚持使用干胎，会遭遇巨额的时间惩罚（每圈慢 5~25 秒不等，具体视积水程度而定）。
  4. **随机事故 (DNF)**：包含引擎故障与撞车的随机性检测。
- **`evaluate_ai_strategy` (AI 策略大脑)**：非玩家 (NPC) 车手的全自动换胎判决模块。它会根据剩余圈数、前车差距 (Undercut)、当前天气以及是否有安全车出动等信息，自动决定最优进站策略。**玩家车手由于通过外部 API 控制，会直接跳过此段逻辑。**

### B. `F1Race` (赛事发车与统筹类)
负责管理全场状态、天气系统的更迭、安全车调度以及控制状态机。

- **核心环境变量**：`sc_active` (安全车是否激活)、`weather_state` (天气大类：Dry/Raining/Drying)、`track_wetness` (绝对赛道湿润度：0.0 到 1.0)。
- **`try_trigger_event` (赛事中断检测器)**：在每一圈计算之前，引擎会检查玩家车手是否踩中了以下三种“具有戏剧性的决策点”：
  1. `SC_PIT`：安全车刚刚出动。系统提示：“要不要趁现在进站蹭一个廉价换胎？”
  2. `WEATHER_PIT`：突然降雨或赛道跨越湿地阈值 (0.2)。系统提示：“由于降雨增加，是否考虑更换雨地胎？”
  3. `TIRE_CLIFF`：玩家的当前轮胎寿命耗尽到极限边界的 85%。系统提示：“轮胎已严重衰减，是否进站？”
- **`advance_until_event` (仿真主循环)**：
  - **恢复与防死锁设计 (`_mid_lap_pause`)**：为防止玩家在同一圈回答“否”后陷入死循环，引擎采用了精准的标记恢复设计。暂停恢复后将跳过本圈“天气更新”阶段，直接恢复赛车行进。
  - **安全车调度逻辑**：当捕捉到 DNF (退赛) 生成时，引擎有 50% 概率出动全场安全车将赛车序列压缩（即间隔重置为 0.6 秒），另外 50% 仅触发不影响整体圈速的“局部黄旗”以贴近现实。

## 3. 随机性与数值平衡 (Probabilities & Balancing)

为确保游戏体验贴近真实的 F1 赛季节奏，避免比赛变得繁乱，内置参数做了如下把控：
1. **降雨概率**：基础单圈降雨突发率被控制在千分之六 (`0.006`)。保证了在 50 圈比赛中，遇到雨战的整体数学期望约为适宜的 **20%~25%** 左右。
2. **退赛 (DNF) 几率**：
   - 基础机械故障与干地碰撞均被下压到极低概率水平（单圈单车不足万分之五），防止每场比赛出现大翻车。
   - 保留了“雨天未换雨胎的高风险概率”，强制建立不进站惩罚；若妥善换上雨胎，则事故几率恢复正常。

## 4. API 运行交互回路 (API Application Lifetime)

配合 `app.py` 内部署的 FastAPI 服务，前端通过 3 个接口完美接入这套核心。

1. **`POST /api/start`** 
   - 动作：加载20位车手网格（Grid），设置玩家焦点，初始化全局状态并返回原始排位。
2. **`POST /api/advance`** 
   - 动作：开始静默推演赛程。不断增加 `lap` 指针直到完成 50 圈或被 `try_trigger_event` 拦截。
   - 返回状态：`"FINISHED"`（完毕）或 `"PAUSED"`（被中断等待玩家选择）。
3. **`POST /api/decision`** 
   - 动作：收到玩家对当前突发事件的表态（决定是否 `Box box` 和具体想要换什么胎）。利用目标指令对 `driver.pit_stop()` 进行结算后，紧接着递归调用 `advance_until_event` 继续剩余跑圈。
